# 어드민 관리 콘솔 — 플랜 (초안)

> 운영자가 사용자·사용량·안전·LLM 운영·지표를 관리하는 웹 콘솔. 모바일 앱과 별개 surface.
> 상태: **플랜 / 결정 확정**(2026-06-19). 호스팅 = **새 `apps/admin`(React Vite)**, 인증 = **Firebase custom claim `admin`**.

## 1. 목적

- 서비스 상태를 한눈에 본다(지표·사용량).
- 사용자·분석권을 운영한다(조회·수동 지급/차감).
- **안전**을 지킨다(조작·압박 등 규칙 위반 의심 분석 검토).
- **LLM 운영**(provider·품질·비용)을 관리한다.

## 2. 핵심 원칙 (이 제품 특수성)

- **원문 미저장 준수**([AGENTS.md](../../AGENTS.md)): 어드민도 대화 원문을 보지 않는다. **집계·구조화 데이터만** 노출(요약·상태·전략·outcome·점수). 개인 식별 정보 표시 금지(별칭·비식별 userId만).
- **최소 권한**: 어드민 전용 인증 + 역할(RBAC). 모든 쓰기 작업은 **audit log**.
- **안전 우선**: "조작/압박/집착 조장" 출력이 나오지 않는지 모더레이션 뷰로 점검(제품 포지션 = 코치, 픽업툴 아님).
- **mock-first**: 어드민도 mock/집계 데이터로 화면을 먼저 완성하고, 실데이터·키는 나중에.

## 3. 호스팅·아키텍처 — ★결정 필요★

| 안 | 내용 | 장 | 단 |
| --- | --- | --- | --- |
| **A. `apps/web` 재사용** | 기존 React(Vite) 프로토타입에 `/admin` 라우트 | 새 프로젝트 0, 빠름 | 사용자용 웹과 빌드/권한 섞임 |
| **B. 새 `apps/admin`** | 별도 React(Vite) admin SPA | 관심사 분리·권한 격리 깔끔 | 새 workspace 셋업 |
| **C. 서버 렌더(Spring + Thymeleaf)** | 백엔드가 admin 페이지 직접 렌더 | 인증 일원화, SPA 불필요 | 프론트 표현력↓ |

→ **추천: B(새 `apps/admin`)** — 권한·배포 격리가 운영툴엔 안전. (소규모면 A도 합리적)

**공통 백엔드**: 어드민 전용 API를 별도 경로(`/api/admin/**`)로 두고 RBAC로 보호. 새 `admin` bounded context 또는 각 컨텍스트에 admin 조회 추가.

## 4. 인증·인가 — ★결정 필요★

- 일반 사용자는 Firebase(익명/소셜). 어드민은 **분리**해야 함.
- 안:
  1. **Firebase custom claim `admin`** — 기존 인증 재사용 + claim으로 게이트(`@PreAuthorize("hasRole('ADMIN')")`). 추천.
  2. 별도 admin 계정 테이블 + 자체 로그인.
- 공통: `/api/admin/**`는 ADMIN 역할 필수, 모든 변경은 audit log(누가·언제·무엇).

## 5. 화면/기능 (MVP)

1. **대시보드** — 핵심 지표: DAU/WAU, 분석 수, 저장률, **outcome 분포**(보냈고 좋음/그냥/안보냄), 분석권 사용·잔여, 리워드 시청. (PMF·리텐션 신호)
2. **사용자** — 목록(비식별 userId, 온보딩 여부, 가입일, 사용량) + 상세(프로필 설정값·성향 점수 등 비식별, 분석권 ledger). 원문·실명 없음.
3. **분석권/Ledger** — 충전·차감 이력, **수동 지급/조정**(CS 대응), 무료/리워드 한도 설정.
4. **안전·모더레이션** — 안전 규칙 위반 의심 분석 플래그 큐(자동 탐지 규칙 + 수동 검토), 처리 상태.
5. **LLM 운영** — 현재 provider/모델, **품질 비교 리포트**(이미 있는 `analysis-quality` fixture/리포트 연계), 토큰·비용 모니터.
6. **콘텐츠/운영** — 공지·FAQ(있다면), mock 시나리오 점검, 기능 플래그.
7. **상담방·분석 통계** — 상대별/전략별/시나리오별 **집계만**(원문 X).

## 6. 백엔드 (admin API)

- 경로: `/api/admin/...` (RBAC ADMIN).
- 신규 조회: 사용자 목록/상세, ledger, 집계 통계, 안전 플래그, 지표.
- 신규 변경: 분석권 수동 지급/차감(audit), 한도 설정, 플래그 처리, 기능 플래그.
- 기존 컨텍스트(identity/credit/analysis/consultation/operations) 재사용 + 집계 쿼리 추가. `operations` 컨텍스트가 admin/운영의 자연스러운 자리.

## 7. mock-first 적용

- admin도 단일 seam: 지표·목록은 **mock 집계 데이터**로 먼저 렌더 → 화면·흐름 완성 → 실 쿼리 연결.
- 안전 자동탐지·LLM 비용 등 실데이터 의존 항목은 mock 후 나중에 실연결.

## 8. Phase 계획

- **Phase 1 (운영 최소)**: 인증·RBAC + 대시보드(핵심 지표) + 사용자 목록/상세 + 분석권 ledger·수동 지급. mock 데이터.
- **Phase 2 (안전·품질)**: 모더레이션 큐 + LLM 운영(품질 리포트 연계).
- **Phase 3 (성장·수익)**: 콘텐츠/플래그 + 결제 지표(결제 연동 후).

## 9. 결정 (확정 2026-06-19)

1. ✅ **호스팅 = 새 `apps/admin`** (React Vite, 권한·배포 격리).
2. ✅ **어드민 인증 = Firebase custom claim `admin`** + `/api/admin/**` RBAC(ADMIN) + audit.
3. 1차 범위: **Phase 1**(인증 + 대시보드 + 사용자 + 분석권 ledger)부터, mock-first.

## 10. 위험

- **프라이버시 사고**: admin이 실수로 원문/식별정보 노출 → 설계 단계에서 "집계·비식별만" 강제(DTO에 원문 필드 없음).
- **권한 상승**: admin API 보호 미흡 → RBAC + audit + 별도 origin.
- **범위 팽창**: 운영툴은 무한 확장 → Phase 1로 좁게 시작.
