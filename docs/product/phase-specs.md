# 플러팅지옥 페이즈별 스펙

## 목적

이 문서는 개발 페이즈별로 실제 구현해야 할 스펙을 정의한다.

`development-phases.md`가 로드맵이라면, 이 문서는 각 페이즈에서 개발자가 확인해야 할 요구사항과 완료 기준이다.

## Phase 1 스펙. 웹 MVP 화면

### 사용자 스토리

- 사용자는 카톡/DM 내용을 붙여넣고 현재 분위기를 알고 싶다.
- 사용자는 지금 보내도 되는 답장을 빠르게 복사하고 싶다.
- 사용자는 자기 말투와 이상형 기준이 반영된 답장을 원한다.
- 사용자는 개인정보가 저장되는지 불안하므로 저장 경계를 알고 싶다.

### 기능 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| `WEB-001` | 첫 화면에 앱 이름과 핵심 카피를 표시한다. | P0 |
| `WEB-002` | 최근 대화를 붙여넣는 textarea를 제공한다. | P0 |
| `WEB-003` | 관계 단계, 대화 목적, 답장 강도, 조언 수위를 선택할 수 있다. | P0 |
| `WEB-004` | 이상형/연애 스타일을 선택할 수 있다. | P0 |
| `WEB-005` | 분석 버튼 클릭 시 API에 분석 요청을 보낸다. | P0 |
| `WEB-006` | 분석 중 상태를 표시한다. | P0 |
| `WEB-007` | 현재 분위기, 적합도, 말투 분석, 답장 후보, 위험한 말을 표시한다. | P0 |
| `WEB-008` | 답장 후보를 복사할 수 있다. | P0 |
| `WEB-009` | 무료 분석 잔여 횟수를 표시한다. | P1 |
| `WEB-010` | 모바일 화면에서 주요 버튼은 최소 44px 터치 영역을 가진다. | P0 |

### 비기능 요구사항

- 빌드 명령 `npm run build`가 성공해야 한다.
- 첫 로딩 화면은 모바일에서 깨지지 않아야 한다.
- 브라우저에 AI API 키를 저장하지 않는다.
- 원문 대화 저장 여부에 대한 안내가 입력창 근처에 있어야 한다.

### 완료 기준

- 사용자가 샘플 대화로 분석 결과를 볼 수 있다.
- 답장 후보 3개가 표시된다.
- 답장 복사 버튼이 동작한다.
- 타입체크와 웹 빌드가 통과한다.

## Phase 2 스펙. Workers API + D1

### 사용자 스토리

- 사용자는 하루 무료 3회까지 분석을 사용할 수 있다.
- 사용자는 남은 무료 분석 횟수를 확인할 수 있다.
- 운영자는 사용자가 분석을 시작/완료/복사했는지 이벤트로 확인할 수 있다.

### API 요구사항

| ID | Endpoint | 요구사항 | 우선순위 |
|---|---|---|---|
| `API-001` | `GET /api/health` | API 상태를 반환한다. | P0 |
| `API-002` | `GET /api/usage` | 오늘 사용량과 남은 무료 횟수를 반환한다. | P0 |
| `API-003` | `POST /api/analyses` | 입력값을 검증하고 분석 결과를 반환한다. | P0 |
| `API-004` | `POST /api/analyses` | 하루 3회 초과 시 `LIMIT_REACHED`를 반환한다. | P0 |
| `API-005` | `POST /api/events` | 답장 복사/피드백 이벤트를 저장한다. | P1 |

### DB 요구사항

| ID | 테이블 | 요구사항 | 우선순위 |
|---|---|---|---|
| `DB-001` | `anonymous_users` | 익명 사용자 ID와 마지막 방문 시각을 저장한다. | P0 |
| `DB-002` | `usage_days` | `anonymous_user_id + usage_date` 기준 사용량을 저장한다. | P0 |
| `DB-003` | `events` | 주요 사용자 행동과 시스템 이벤트를 저장한다. | P0 |
| `DB-004` | `analyses` | 분석 결과 요약을 저장한다. 원문 대화는 저장하지 않는다. | P0 |
| `DB-005` | `user_profiles` | 이상형/말투 요약 프로필을 저장할 수 있다. | P1 |

### 완료 기준

- D1 migration이 성공한다.
- `GET /api/health`가 운영 환경에서 성공한다.
- 1~3번째 분석은 성공한다.
- 4번째 분석은 `LIMIT_REACHED`를 반환한다.
- D1에서 이벤트 row를 확인할 수 있다.

## Phase 3 스펙. 실제 AI 분석

### 사용자 스토리

- 사용자는 실제 대화에 맞는 답장을 받고 싶다.
- 사용자는 자기 말투와 다르게 튀는 답장을 피하고 싶다.
- 사용자는 이상형과 다르더라도 계속 만나고 싶은 마음을 존중받고 싶다.
- 사용자는 부담스럽거나 위험한 말을 피하고 싶다.

### AI 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| `AI-001` | AI 응답은 `ai-response-schema.md` 구조를 따라야 한다. | P0 |
| `AI-002` | 사용자의 말투를 분석하고 답장 후보에 반영해야 한다. | P0 |
| `AI-003` | 답장 후보는 `순한맛`, `설렘맛`, `직진맛` 3개를 제공한다. | P0 |
| `AI-004` | 사용자의 이상형과 상대 스타일 차이를 경고등처럼 설명한다. | P0 |
| `AI-005` | 상대를 만나도 되는지 단정하지 않는다. | P0 |
| `AI-006` | 강압, 조작, 집요한 접근은 안전 안내로 전환한다. | P0 |
| `AI-007` | AI JSON 파싱 실패 시 사용자에게 재시도 안내를 제공한다. | P0 |

### 프롬프트 요구사항

- 시스템 프롬프트는 제품 원칙과 안전 정책을 포함한다.
- 출력 프롬프트는 JSON schema를 강제한다.
- 말투 분석 프롬프트는 사용자 발화만 기준으로 한다.
- 원문 대화는 저장하지 않는다는 정책을 서버 코드와 문서에 맞춘다.

### 완료 기준

- 실제 LLM API 호출이 Workers에서만 일어난다.
- 브라우저에 LLM API 키가 노출되지 않는다.
- 샘플 대화 10개에서 응답 JSON 구조가 모두 유효하다.
- 안전하지 않은 요청 샘플에서 답장 추천 대신 경고가 반환된다.

## Phase 4 스펙. 비공개 베타 검증

### 사용자 스토리

- 초기 사용자는 실제 대화로 분석을 써보고, 답장이 도움이 되는지 판단한다.
- 운영자는 어떤 화면에서 이탈하는지 확인한다.
- 운영자는 AI 답장 품질을 개선할 근거를 모은다.

### 기능 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| `BETA-001` | 분석 완료 이벤트를 기록한다. | P0 |
| `BETA-002` | 답장 복사 이벤트를 기록한다. | P0 |
| `BETA-003` | `도움 됨`, `별로임` 피드백을 기록한다. | P0 |
| `BETA-004` | AI 실패율과 제한 도달률을 볼 수 있게 이벤트를 남긴다. | P1 |

### 완료 기준

- 최소 10명의 테스트 사용자가 실제 대화로 사용한다.
- 답장 복사율을 계산할 수 있다.
- 반복되는 불만 3개 이상을 정리한다.
- 다음 개선 우선순위를 정한다.

## Phase 5 스펙. 분석권 결제

### 사용자 스토리

- 사용자는 무료 분석을 모두 사용한 뒤 분석권을 구매할 수 있다.
- 사용자는 결제 후 잔액이 바로 반영되기를 기대한다.
- 운영자는 결제 webhook 중복 수신에도 크레딧이 중복 지급되지 않게 해야 한다.

### 기능 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| `PAY-001` | 분석권 패키지 목록을 표시한다. | P0 |
| `PAY-002` | Workers에서 Polar Checkout을 생성한다. | P0 |
| `PAY-003` | Polar webhook을 수신한다. | P0 |
| `PAY-004` | 결제 완료 시 credit ledger에 지급 기록을 남긴다. | P0 |
| `PAY-005` | 같은 webhook event는 한 번만 처리한다. | P0 |
| `PAY-006` | 분석 1회 사용 시 유료 크레딧을 1 차감한다. | P0 |

### 완료 기준

- 테스트 결제 완료 후 분석권이 증가한다.
- webhook 재전송에도 잔액이 중복 증가하지 않는다.
- 무료 횟수 초과 시 구매 안내가 표시된다.
- 잔액이 있을 때는 무료 횟수 초과 후에도 분석이 가능하다.

## Phase 6 스펙. PWA/앱 확장

### 사용자 스토리

- 사용자는 플러팅지옥을 홈 화면에 앱처럼 추가하고 싶다.
- 사용자는 모바일에서 매번 URL을 찾지 않고 빠르게 열고 싶다.

### 기능 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| `PWA-001` | Web App Manifest를 추가한다. | P0 |
| `PWA-002` | 앱 아이콘을 추가한다. | P0 |
| `PWA-003` | 기본 service worker 또는 offline fallback을 추가한다. | P1 |
| `PWA-004` | 홈 화면 추가 안내를 제공한다. | P1 |
| `PWA-005` | 앱스토어 배포 필요성은 사용 지표로 판단한다. | P1 |

### 완료 기준

- 모바일 브라우저에서 홈 화면 추가가 가능하다.
- 홈 화면 아이콘과 앱 이름이 정상 표시된다.
- PWA가 웹 MVP 기능을 깨뜨리지 않는다.

## 공통 검증 기준

모든 페이즈는 다음 기준을 통과해야 완료로 본다.

```text
npm run typecheck
npm run build
```

API 변경이 있는 페이즈는 추가로 다음을 확인한다.

```text
npm run build:api
```

## 우선순위 정의

| 우선순위 | 의미 |
|---|---|
| `P0` | 페이즈 완료에 반드시 필요 |
| `P1` | 있으면 좋지만 페이즈 완료를 막지 않음 |
| `P2` | 다음 페이즈 또는 후순위 개선 |
