# Mock-First 개발 전략

> 목표: **실 LLM 없이(키 없이) 전 기능을 완성**하고, 마지막에 키 1회 투입으로 품질만 검증·전환한다.

## 0. 한 줄 원칙

LLM이 사는 곳을 **단 하나의 seam**(`AnalysisPort`)으로 가둔다. 그 외 모든 코드(화면·플로우·영속화·메모리·개인화·적합도·결제구조)는 LLM을 모른 채 동작한다.

```
[모든 화면·서비스·영속화·메모리·개인화]  ← provider-무관, 항상 결정적
                 │
          AnalysisPort.analyze()         ← LLM은 오직 여기
            ├─ MockAnalysisAdapter   (기본값, 개발 내내 사용)
            └─ LlmAnalysisAdapter    (키 줄 때만, 출시/품질검증용)
```

전환은 코드 변경 없이 환경변수 하나:
```bash
FLIRTING_HELL_AI_PROVIDER=mock     # 기본 — 키 불필요
FLIRTING_HELL_AI_PROVIDER=claude   # 최종 — 키 투입
```

## 1. 4가지 규칙 (이걸 지키면 mock으로 끝까지 간다)

1. **단일 seam.** 새 AI 기능(챗 코칭·데이트 플랜·장소 추천)도 *반드시* 포트 인터페이스 뒤에 둔다. 화면/서비스가 LLM 클라이언트를 직접 호출하지 않는다.
2. **mock = 계약 완전 일치.** mock은 실 LLM이 채울 **모든 필드**를 유효 범위로 채운다. → 실 LLM에서만 보이는 빈칸이 생기지 않는다. (`MockAnalysisParityTest`가 이를 강제)
3. **시나리오 기반 mock.** 입력 신호로 상황(호감/약속/회복/거리두기/일상)을 감지해 상대 5축 유형·요약·주의가 현실적으로 달라진다. → 키 없이 데모·QA·스크린샷·사용자 테스트 가능.
4. **결정적.** 같은 입력 → 같은 출력. 테스트가 흔들리지 않는다(실 LLM은 매번 달라 테스트가 어렵다).

## 2. 지금 mock으로 검증되는 것 (키 0회)

- 전체 화면·플로우, 상담방 영속화, 분석권 구조.
- 성향 계정 영속화, 상대 5축 유형 추론, 이상형 적합도.
- **대화 연속성**(이전 요약 주입) + **답장 개인화**(상대 유형 차이 반영).
- 위 전부 backend `gradlew test` + mobile `flutter test` + 실DB e2e로 통과.

## 3. mock이 못 하는 단 하나

**조언의 실제 품질·안전성.** mock은 "흐름이 도는지"는 증명하지만 "AI 답장이 진짜 좋은지/위험하지 않은지"는 검증 못 한다. → 키는 *개발용*이 아니라 **완성 후 품질검증·출시용**으로만 1회 투입.

## 4. 완성 시 LLM 전환 체크리스트

1. `LlmAnalysisAdapter.responseSchema()`/응답 record에 `partnerType`(5축) 추가.
2. `userPrompt`는 이미 성향·이전 요약·상대유형 맥락 + 개인화 지시 포함(완료).
3. `ai-analysis-contract.md`에 partnerType 명시.
4. 키·provider 환경변수 설정 후 `test:backend:analysis-quality:real`로 품질 비교.
5. 안전 규칙(단정·압박 금지) 출력 점검.

## 5. 새 AI 기능을 mock-first로 추가하는 법

1. 기능의 **출력 계약**(record/JSON 스키마)을 먼저 정의.
2. 포트 인터페이스 + **Mock 어댑터(결정적·시나리오 기반)** 부터 구현.
3. 화면/서비스/영속화/테스트를 mock으로 완성.
4. 패리티 테스트로 "mock이 계약 전 필드 채움" 고정.
5. 마지막에 LLM 어댑터의 프롬프트/파싱만 채워 전환.
